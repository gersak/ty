Button Component Styling Approaches

Approach 1: Inline Styles (Original)

✅ Pros:

  • No external dependencies
  • Styles are always available
  • Works offline

❌ Cons:

  • Styles mixed with logic
  • Harder to maintain
  • Duplicated for each instance

Approach 2: External CSS File

✅ Pros:

  • Clean separation of concerns
  • Easy to edit styles
  • Can be cached by browser

❌ Cons:

  • Extra HTTP request per component type
  • Path must be correct
  • Might have CORS issues
<link rel="stylesheet" href="/dev/styles/button.css">

Approach 3: CSS as String Constant

✅ Pros:

  • Styles in separate constant/file
  • No external requests
  • Works offline

❌ Cons:

  • Style tag duplicated for each instance
  • Slightly larger bundle size

Approach 4: Constructable Stylesheets (Recommended)

✅ Pros:

  • Most efficient - single stylesheet shared across all instances
  • No DOM duplication
  • Fast updates
  • Clean separation

❌ Cons:

  • Requires modern browser support
  • Slightly more complex setup
// Create once
const sheet = new CSSStyleSheet();
sheet.replaceSync(cssText);

// Use in each instance
shadowRoot.adoptedStyleSheets = [sheet];

Build-Time CSS Loading

For production, you can load CSS at build time using macros:

(defmacro load-css [path]
  (slurp (io/resource path)))

(def button-styles (load-css "styles/button.css"))

This gives you the benefits of separate CSS files during development while bundling them at build time.
